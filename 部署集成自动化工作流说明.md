# 部署集成自动化工作流说明

> 本文档详细记录个人网站从代码托管到自动化部署的完整流程、问题排查与解决方案，适用于 Vite SSG 静态网站 + 自有服务器的部署场景。

## 一、整体架构

```
┌─────────────────┐
│  本地开发环境    │
│  (Mac/Linux)    │
└────────┬────────┘
         │ git push
         ▼
┌─────────────────┐
│   GitHub 仓库    │
│  qinxiushan/    │
│  personal-      │
│  website        │
└────────┬────────┘
         │ 触发 workflow
         ▼
┌─────────────────┐
│ Self-hosted     │
│ Runner (服务器)  │
│ - git pull      │
│ - pnpm install  │
│ - pnpm build    │
│ - reload nginx  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Nginx 服务器    │
│  <YOUR_SERVER_IP>  │
│  /dist → 静态文件│
└─────────────────┘
```

**核心原则**：所有构建和部署操作都在服务器上完成，避免 GitHub Actions 托管 Runner 的网络限制问题。

## 二、为什么选择 Self-hosted Runner

### 2.1 尝试过的方案

| 方案                                                  | 问题                                                    |
| ----------------------------------------------------- | ------------------------------------------------------- |
| **GitHub Actions + SSH** (ubuntu-latest runner) | 托管 Runner IP 在美国，被服务器安全策略拦截（高危预警） |
| **GitHub Actions + SCP**                        | 同样需要从 GitHub 推送到服务器，被拦截                  |
| **Docker 镜像推送**                             | 需要镜像仓库，增加复杂度                                |

### 2.2 最终方案：Self-hosted Runner

**原理**：在目标服务器上安装 GitHub Actions Runner，GitHub 通过 HTTPS 协议发送任务，Runner 在服务器本地执行脚本（git pull、构建等），完全绕过 SSH 远程连接。

**优势**：

- 不需要 SSH 远程登录服务器
- Runner 在服务器本地运行，git pull 不受网络限制
- 资源消耗可控（构建只在服务器上发生）
- 触发方式是 HTTPS 协议，不会被 IP 拦截

## 三、完整部署流程

### 3.1 服务器环境准备

#### 3.1.1 安装基础环境

```bash
# 安装 Node.js 22（项目要求）
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs

# 安装 pnpm 11.2.2（使用 corepack 锁定版本）
corepack enable
corepack prepare pnpm@11.2.2 --activate

# 安装 Nginx
sudo apt install -y nginx
```

**什么是 corepack？为什么不直接 `npm install -g pnpm`？**

| 方式 | 行为 | 风险 |
|---|---|---|
| `npm install -g pnpm` | 装最新版本 | 可能与 `package.json` 的 `packageManager: pnpm@11.2.2` 不匹配 |
| `corepack prepare pnpm@11.2.2 --activate` | 锁定到指定版本 | 与项目完全一致 |

**corepack** 是 Node.js 官方内置的包管理器管理工具（Node 16.10+），用来管理 pnpm/yarn 的版本。它能读取 `package.json` 里的 `packageManager` 字段，自动切换到正确版本。

**为什么会遇到 `styleText` 报错？** 之前的错误是因为 `npm install -g pnpm` 装了最新版（如 pnpm 12+），而新版 pnpm 的依赖 `listr2` 用了 Node 21.7+ 才有的 `styleText` API，Node 18 直接报 `SyntaxError`。

**替代方案**：如果不想用 corepack，也可以 `npm install -g pnpm@11.2.2` 指定版本。

#### 3.1.2 创建项目目录并克隆代码

```bash
mkdir -p <YOUR_PARENT_DIR>
cd <YOUR_PARENT_DIR>
git clone git@github.com:<USER>/<REPO>.git chengjiabiao
cd chengjiabiao
```

**为什么手动克隆一次**：Self-hosted Runner 第一次执行时目录已存在，避免 runner 内部处理 git clone 的复杂性。

**注意**：本示例使用占位符 `<YOUR_*>`，生产环境请使用真实路径但不要在公开文档中暴露。

### 3.2 Self-hosted Runner 安装

#### 3.2.1 下载 Runner

```bash
mkdir -p ~/actions-runner && cd ~/actions-runner

# 下载（国内服务器可能无法直连 GitHub，需本地下载后 scp）
curl -L -o actions-runner-linux-x64-2.319.1.tar.gz \
  https://github.com/actions/runner/releases/download/v2.319.1/actions-runner-linux-x64-2.319.1.tar.gz
```

**坑点 1**：服务器在中国，curl GitHub 经常失败（`Failure when receiving data from the peer`）。
**解决方案**：在本地电脑下载后 scp 上传。

```bash
# 本地下载
curl -L -o actions-runner-linux-x64-2.319.1.tar.gz \
  https://github.com/actions/runner/releases/download/v2.319.1/actions-runner-linux-x64-2.319.1.tar.gz

# scp 到服务器
scp actions-runner-linux-x64-2.319.1.tar.gz <USER>@<YOUR_SERVER_IP>:~/actions-runner/
```

#### 3.2.2 解压并配置

```bash
tar xzf actions-runner-linux-x64-2.319.1.tar.gz

# 配置 Runner（需要 token）
./config.sh --url https://github.com/qinxiushan/personal-website --token YOUR_TOKEN
```

**坑点 2**：Token 在 https://github.com/qinxiushan/personal-website/settings/actions/runners/new 页面生成，一次性有效，过期需重新生成。

#### 3.2.3 安装为系统服务

```bash
sudo ./svc.sh install
sudo ./svc.sh start
sudo ./svc.sh status
```

**为什么必须用 svc.sh**：直接运行 `./run.sh` 是前台模式，关闭终端就停止，CI 任务无法触发。svc.sh 创建 systemd 服务，开机自启、后台运行。

### 3.3 GitHub Secrets 配置

在 https://github.com/qinxiushan/personal-website/settings/secrets/actions 添加：

| Secret 名称        | 值                | 作用                                  |
| ------------------ | ----------------- | ------------------------------------- |
| `SERVER_HOST`    | `<YOUR_SERVER_IP>` | 服务器 IP（本方案暂未使用，保留备用） |
| `SERVER_USER`    | `ubuntu`        | SSH 用户名（备用）                    |
| `SERVER_SSH_KEY` | 私钥内容          | SSH 私钥（备用）                      |

**本方案（Self-hosted Runner）实际上不需要这些 secrets**，因为 Runner 已经在服务器上，不需要 SSH。

### 3.4 Nginx 配置

#### 3.4.1 创建站点配置

```bash
sudo nano /etc/nginx/sites-available/chengjiabiao
```

```nginx
server {
    listen 80;
    server_name chengjiabiao.com www.chengjiabiao.com <YOUR_SERVER_IP>;

    root <YOUR_PROJECT_PATH>/dist;
    index index.html;

    # 静态资源缓存
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    # SPA 路由回退
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

#### 3.4.2 启用站点

```bash
sudo ln -s /etc/nginx/sites-available/chengjiabiao /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t
sudo systemctl restart nginx
```

**坑点 3**：必须删除默认的 `sites-enabled/default`，否则会冲突导致新配置不生效。

**坑点 4**：`try_files $uri $uri/ /index.html;` 是 SPA 路由的关键，刷新页面时不会 404。

### 3.5 GitHub Actions Workflow

文件位置：`.github/workflows/deploy.yml`

```yaml
name: Deploy to Server

on:
  push:
    branches: [main]
  workflow_dispatch:  # 允许手动触发

jobs:
  deploy:
    runs-on: self-hosted
    steps:
      - name: Deploy
        run: |
          cd <YOUR_PROJECT_PATH>
          git pull origin main
          pnpm install --frozen-lockfile
          pnpm run build
          sudo systemctl reload nginx
```

**关键点**：

- `runs-on: self-hosted` → 使用服务器上的 Runner
- `pnpm install --frozen-lockfile` → 保证依赖版本一致
- `sudo systemctl reload nginx` → 不中断服务的配置重载（比 restart 更安全）

## 四、踩过的坑汇总

| # | 问题                                          | 原因                        | 解决方案                           |
| - | --------------------------------------------- | --------------------------- | ---------------------------------- |
| 1 | `styleText` not exported from `node:util` | Node.js v18.19.1 太老       | 升级到 Node.js 22                  |
| 2 | curl GitHub 失败                              | 国内服务器访问 GitHub 受限  | 本地下载后 scp                     |
| 3 | `appleboy/ssh-action` timeout               | 托管 Runner 无法 SSH 服务器 | 改用 Self-hosted Runner            |
| 4 | 服务器安全策略拦截 GitHub IP                  | GitHub Runner IP 在美国     | Self-hosted Runner 绕过            |
| 5 | Runner 接收任务后没反应                       | 没安装为系统服务            | `sudo ./svc.sh install && start` |
| 6 | Nginx 配置不生效                              | 默认站点冲突                | 删除`sites-enabled/default`      |
| 7 | 刷新页面 404                                  | 缺少 SPA fallback           | 添加`try_files`                  |
| 8 | runner 任务超时 10 分钟                       | 构建+安装依赖慢             | 拆步骤或预先装好依赖               |

## 七、安全风险与加固

### 7.1 Self-hosted Runner 的安全风险

GitHub 在添加 Runner 页面会强制显示警告：

> **Using self-hosted runners in public repositories is not recommended.**  
> Forks of your public repository can potentially run dangerous code on your self-hosted runner by creating a pull request.

**风险原理**：
1. 你的仓库是 public → 任何人都能 fork
2. 外部 PR 触发 workflow → 如果 workflow 没限制事件类型，会在 Runner 上执行
3. Runner 跑在你自己的服务器上 → 攻击者拿到 RCE（远程代码执行）

**我们方案的安全保障**：

```yaml
on:
  push:
    branches: [main]   # 只监听 main 分支的 push
  workflow_dispatch:    # 允许手动触发
  # 故意不写 pull_request，避免 PR 触发
```

- **不监听 `pull_request` 事件** → 外部 PR 不会触发部署
- **只监听 `main` 分支 push** → 只有仓库管理员（你）能 push
- **配合 GitHub 分支保护规则** → 即使有人拿到 push 权限也需要审核

**进一步加固（推荐）**：

1. **创建专用 Runner 用户**（不用 root/ubuntu 跑 Runner）

```bash
sudo useradd -m -s /bin/bash runner
sudo usermod -aG sudo runner  # 限制 sudo 权限
sudo -u runner bash
# 在 runner 用户下安装并配置 runner
```

2. **限制 sudo 权限**

```bash
# /etc/sudoers.d/runner
runner ALL=(ALL) NOPASSWD: /usr/bin/systemctl reload nginx
runner ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx
```

3. **配置分支保护规则**

仓库 → Settings → Branches → Add rule:
- Branch name pattern: `main`
- ☑ Require a pull request before merging
- ☑ Require approvals: 1

4. **定期审计 Runner 日志**

```bash
journalctl -u actions.runner.* --since "1 day ago"
```

### 7.2 文档/博客暴露信息的风险

**绝不能在公开文档中暴露**：
- ❌ 服务器真实 IP
- ❌ SSH 用户名
- ❌ 项目绝对路径
- ❌ 真实的 Runner token（一次性，但泄露也不好）
- ❌ 任何 secrets 的值

**正确做法**：
- ✅ 用占位符：`<YOUR_SERVER_IP>`、`<YOUR_PROJECT_PATH>`、`<USER>`、`<REPO>`
- ✅ 真实配置写在私有笔记（Notion 本地版、本地 Markdown）
- ✅ 截图时打码敏感信息
- ✅ 配置文件用环境变量，不用硬编码

### 7.3 服务器侧防护清单

| 防护项 | 命令/配置 | 重要性 |
|---|---|---|
| 改 SSH 端口 | `/etc/ssh/sshd_config`: `Port 2222` | ⭐⭐⭐ |
| 禁用密码登录 | `PasswordAuthentication no` | ⭐⭐⭐⭐⭐ |
| 仅允许密钥登录 | `PubkeyAuthentication yes` | ⭐⭐⭐⭐⭐ |
| 安装 fail2ban | `sudo apt install fail2ban` | ⭐⭐⭐⭐ |
| 配置 UFW 防火墙 | `sudo ufw allow 2222/tcp` | ⭐⭐⭐⭐ |
| 限制 sudo 用户 | `/etc/sudoers` 配置 | ⭐⭐⭐ |
| 定期更新系统 | `sudo apt update && sudo apt upgrade` | ⭐⭐⭐⭐ |
| Nginx 隐藏版本 | `server_tokens off;` | ⭐⭐ |
| 限制敏感路径 | Nginx `internal;` 指令 | ⭐⭐⭐ |
| 启用 HTTPS | Let's Encrypt + certbot | ⭐⭐⭐⭐⭐ |

### 7.4 HTTPS 配置

```bash
# 安装 certbot
sudo apt install certbot python3-certbot-nginx

# 自动申请证书并配置 Nginx
sudo certbot --nginx -d chengjiabiao.com -d www.chengjiabiao.com

# 测试自动续期
sudo certbot renew --dry-run
```
